# Data Migration: `clinic/main` เอกสารเดียว → หลาย Collection

## ทำไมต้องย้าย

ตอนนี้ข้อมูลทั้งคลินิก (ผู้ป่วยทุกคน, visit ทุก visit, ผู้ใช้ทุกคน ฯลฯ) เก็บอยู่ใน
เอกสาร Firestore เดียว (`clinic/main`) ทุกครั้งที่เปิดแอพต้องโหลดข้อมูล **ทั้งหมด** มาที่
เครื่อง และทุกการแก้ไข (แม้แค่ 1 visit) ต้องเขียนทับเอกสารทั้งก้อน ปัญหา:

- โหลดช้าลงเรื่อยๆ ตามจำนวนผู้ป่วย/visit ที่เพิ่มขึ้น
- เสี่ยงชนขีดจำกัดขนาดเอกสาร 1MB ของ Firestore เมื่อข้อมูลโตพอ
- บังคับสิทธิ์ (RBAC) ที่ระดับฐานข้อมูลจริงไม่ได้ เพราะ role ของผู้ใช้เก็บอยู่ใน
  เอกสารเดียวกับข้อมูล clinical ที่ทุกคนต้องเขียนร่วมกัน

โครงสร้างเป้าหมาย: แยกเป็น `patients/{id}`, `patients/{id}/visits/{id}`,
`telepharmacy/{id}`, `users/{uid}`, `clinicDayRosters/{date}`, `queueWalkIns/{id}`,
`haCompliance/main` — แต่ละอย่างโหลด/เขียนแยกกันได้

## สถานะปัจจุบัน (สำคัญ — อ่านก่อน)

**ค่าเริ่มต้นของแอพจริง (RxClinic.html) ยังคงอ่าน/เขียน `clinic/main` เอกสารเดียวเหมือนเดิม
ทุกประการ สำหรับผู้ใช้ทุกคน** — เฟส 2 (ด้านล่าง) เขียนเสร็จแล้วในโค้ด แต่ทำงานเฉพาะเมื่อ
admin เปิดโหมด "🧪 โครงสร้างข้อมูลใหม่ (ทดลอง)" จาก Settings ด้วยตัวเองเท่านั้น (opt-in,
default = ปิด) จึงไม่มีผลกับผู้ใช้งานจริงแม้แต่น้อยจนกว่าจะมีคนเปิดใช้เอง

## แผนการย้าย (3 เฟส)

### เฟส 1 — สร้างโครงสร้างใหม่คู่ขนาน (เสร็จแล้ว)

1. `transform.js` — ฟังก์ชันแปลงข้อมูลล้วนๆ ทดสอบแล้วด้วย
   `tests/migration-transform.test.js` (ไม่แตะ Firestore จริงเลย)
2. `migrate.js` — สคริปต์รันจริง อ่าน `clinic/main` แล้วเขียนไปยัง collection ใหม่
   **โดยไม่ลบ/แก้ `clinic/main` เลย** รันซ้ำได้ (idempotent)
3. `firestore.rules.target` — กฎ Firestore สำหรับโครงสร้างใหม่ (ยังไม่ deploy)

### เฟส 2 — แก้แอพให้อ่าน/เขียนโครงสร้างใหม่ (เสร็จแล้ว — แบบ opt-in เท่านั้น)

เพิ่ม adapter ในตัวแอพเอง (`assembleDataFromCollections`, `diffAndWriteChanges` ใน
RxClinic.html — ดู `tests/collections-adapter.test.js` ซึ่งมีเทสต์ตรวจด้วยว่า doc path
ตรงกับ `transform.js` เป๊ะ กันสองระบบหลุด sync กัน) `initFirestore`/`flushToCloud` แตก
สาขาตาม `dataModeRef.current`:
- **`'legacy'` (ค่าเริ่มต้นของทุกคน)** — โค้ดเดิมทุกบรรทัด ไม่ถูกแตะเลย
- **`'collections'` (ทดลอง, เปิดเองจาก Settings → 🧪 โครงสร้างข้อมูลใหม่)** — อ่าน/เขียน
  collection ใหม่แทน โดย**เขียนเฉพาะเอกสารที่เปลี่ยนจริง** (diff เทียบ id) แทนที่จะเขียน
  ก้อนเดียวทั้งหมดเหมือนเดิม

ข้อจำกัดที่ตั้งใจไว้ของโหมดทดลองเวอร์ชันนี้ (ตั้งใจไม่ทำให้ครบ เพื่อลดความเสี่ยง):
ไม่มี realtime listener ข้ามเครื่อง/แท็บ (ต้องรีเฟรชเองเพื่อดูข้อมูลที่คนอื่นแก้), ไม่มี
offline-draft-recovery ที่สมบูรณ์เท่าโหมด legacy — เหมาะกับการทดสอบก่อนตัดสินใจย้ายจริง
ไม่ใช่การใช้งานประจำวัน สลับกลับมาโหมด legacy ได้ทุกเมื่อจาก Settings เดียวกัน

**สิ่งที่ยังต้องทำเอง (ผมไม่มี credential เข้าถึงได้):** รัน `migrate.js --commit` กับ
service account key ของโปรเจกต์จริง แล้วค่อยเปิดโหมดทดลองมาลองดู — ไม่งั้นจะเห็นข้อมูล
ว่างเปล่าเพราะยังไม่มีข้อมูลใน collection ใหม่เลย

### เฟส 3 — ปิด `clinic/main` เดิม / ทำให้ collections เป็นค่าเริ่มต้น (ยังไม่เริ่ม)

ทำได้ก็ต่อเมื่อทดสอบโหมด `'collections'` แล้วมั่นใจ 100% (รวมเติม realtime listener ให้
ครบ) เป็นเวลาพอสมควร — เป็นการตัดสินใจที่ต้องทำร่วมกันตอนนั้น ไม่ใช่ตอนนี้

## วิธีรัน (เฟส 1 เท่านั้น)

```bash
cd migration
npm install          # ติดตั้ง firebase-admin (แยก dependency จากแอพหลัก)

# ขั้น 1: dry-run เสมอก่อน — ปลอดภัย ไม่เขียนอะไรทั้งสิ้น แค่ print แผน+ตรวจนับ
node migrate.js

# ขั้น 2: เมื่อพร้อมเขียนจริง ต้องมี service account key ก่อน:
#   Firebase Console → Project Settings → Service Accounts → Generate new private key
export GOOGLE_APPLICATION_CREDENTIALS=/path/to/serviceAccountKey.json
node migrate.js --commit --project=rxcopdasthmaclinic
```

สคริปต์จะ:
- ปฏิเสธการรัน `--commit` ถ้าไม่ได้ตั้ง `GOOGLE_APPLICATION_CREDENTIALS`
- ตรวจนับจำนวนเอกสารในแผนเทียบกับต้นทางก่อนเขียนจริง (หยุดทันทีถ้าไม่ตรง)
- เขียนเป็น batch (≤400 เอกสาร/batch) ไปยัง collection ใหม่
- อ่านกลับมาตรวจนับหลังเขียนจริงอีกครั้ง (verify)
- **ไม่ลบ/แก้ `clinic/main` ในทุกกรณี** — เป็น fallback เสมอ

## ก่อนรัน `--commit` จริงกับข้อมูล production

- [ ] สำรองข้อมูลด้วยปุ่ม "💾 ดาวน์โหลด Backup ทั้งหมด" ในแอพ (Settings) ก่อนเสมอ
- [ ] รัน `node migrate.js` (dry-run) ดูตัวเลขในแผนให้สมเหตุสมผลกับจำนวนผู้ป่วยจริง
- [ ] แนะนำให้ทดสอบกับโปรเจกต์ Firebase แยกต่างหาก (ไม่ใช่ production) ก่อน ถ้าเป็นไปได้
- [ ] รัน `--commit` แล้วดูผล verify ท้าย log ให้ `ok: true` ทุก collection

## Rollback

**โหมด legacy (ค่าเริ่มต้นของทุกคน)**: เพราะ `clinic/main` ไม่ถูกแตะต้องเลย การ rollback คือ
**ไม่ต้องทำอะไร** — แอพยังอ่าน `clinic/main` เดิมอยู่เสมอ ถ้าพบปัญหาระหว่างรัน migration
แค่ลบ collection ใหม่ที่เขียนไปทิ้ง (หรือปล่อยทิ้งไว้เฉยๆ ก็ได้ ไม่กระทบอะไร เพราะแอพไม่ได้
อ่านมันอยู่ดีตราบใดที่ยังไม่มีใครเปิดโหมดทดลอง) แล้วแก้ปัญหาก่อนรันใหม่

**ถ้าเปิดโหมดทดลอง ('collections') แล้วเจอปัญหา**: กลับไปที่ Settings → 🧪 โครงสร้างข้อมูลใหม่
→ กด "⬅ กลับไปโหมด Legacy" หน้าจะรีโหลดและกลับไปอ่าน `clinic/main` เดิมทันที — ข้อมูลใน
`clinic/main` ไม่เคยถูกแก้ไขระหว่างอยู่โหมดทดลองเลย (เขียนไปที่ collection ใหม่เท่านั้น) จึง
ปลอดภัย 100% ที่จะสลับกลับไปมาได้ตลอด
